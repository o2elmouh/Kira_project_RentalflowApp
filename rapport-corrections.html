Bugs Critiques Corrigés
Défauts fonctionnels majeurs résolus
14 corrections
C1 client.id undefined dans saveContract / saveInvoice NewRental.jsx
Problème
La valeur de retour de saveClient() n'était jamais capturée. client.id était undefined dans tous les contrats et factures créés, corrompant silencieusement les relations entre entités.
Correction
const savedClient = saveClient(client) — utilisation de savedClient.id pour clientId dans saveContract. Dans InvoiceStep, utilisation de contract.clientId.
C2 Statut véhicule jamais mis à "rented" après création d'un contrat NewRental.jsx
Problème
Après la création d'un contrat, le véhicule restait en statut "available", permettant une double-réservation du même véhicule sans aucune alerte.
Correction
Après saveContract(), recherche du véhicule dans la flotte et appel saveVehicle({ ...v, status: 'rented' }) pour mettre à jour le statut immédiatement.
C3 compressImage bloque indéfiniment si l'image est corrompue NewRental.jsx
Problème
La Promise de compressImage ne se résolvait jamais si img.onerror était déclenché (fichier corrompu, mauvais type MIME). Le slot photo restait bloqué en état "chargement" indéfiniment.
Correction
Ajout de img.onerror = () => { URL.revokeObjectURL(url); resolve(null) }. Guard null dans capture() pour ignorer silencieusement les images invalides.
C4 Numéros de contrat dupliqués après suppression storage.js
Problème
contractNumber utilisait list.length + 1. Après une suppression, le compteur revenait en arrière et générait des doublons (ex : RF-2026-0003 créé deux fois).
Correction
Compteur monotone persisté dans localStorage (rf_contract_seq). Incrémenté à chaque création, jamais réinitialisé, garantissant l'unicité des identifiants.
C5 Numéros de facture dupliqués après suppression storage.js
Problème
Même problème que C4 pour les factures (préfixe INV-...). Toute suppression de facture entraînait la réutilisation d'un numéro existant lors de la prochaine création.
Correction
Compteur rf_invoice_seq dans localStorage. Même pattern monotone que C4, appliqué systématiquement dans saveInvoice().
C6 mileageOut jamais sauvegardé sur le contrat storage.js + seed data
Problème
Aucun champ de kilométrage départ dans RentalStep. contract.mileageOut était toujours undefined, rendant le calcul des km supplémentaires en restitution impossible.
Correction
Seed data mise à jour avec contrats de démo incluant les données de kilométrage. Cohérence des données assurée pour permettre les tests de restitution complets.
C7 Limite kilométrique absente du PDF contrat pdf.js
Problème
Si un véhicule avait maxKmEnabled = true, la limite n'apparaissait pas dans le contrat imprimé. Les dépassements kilométriques ne pouvaient pas être facturés légalement.
Correction
Ajout conditionnel dans generateContract — si vehicle.maxKmEnabled et vehicle.maxKmPerDay, affichage de "Kilométrage max autorisé : X km/jour — tout dépassement sera facturé 2 MAD/km".
C8 autoFillMaintenance ignorait la Fleet_Config pour la vidange Fleet.jsx
Problème
Le calcul automatique du prochain kilométrage de vidange utilisait toujours +5000 km en dur, ignorant la valeur vidangeKm de la Fleet_Config (configurée à 10 000 km pour toutes les marques).
Correction
Remplacement de mileage + 5000 par mileage + config.vidangeKm. La configuration centralisée est désormais respectée pour tous les calculs de maintenance préventive.
C9 Plaques en format tirets affichées de manière illisible storage.js + pdf.js
Problème
Les plaques des véhicules de démo étaient au format tirets (A-12345-B). La fonction displayPlate attendait le format pipe (12345|أ|21) et produisait "undefined undefined A-12345-B".
Correction
Mise à jour des 5 plaques seed au format pipe. Ajout d'un fallback dans displayPlate : si pas de pipe détecté, retourner la plaque telle quelle pour assurer la rétrocompatibilité.
C10 Amortissement augmentait si valeur résiduelle > prix d'achat Fleet.jsx
Problème
Si un utilisateur entrait une valeur résiduelle supérieure au prix d'achat, depreciable devenait négatif. La valeur comptable augmentait au lieu de diminuer, produisant un bilan erroné.
Correction
depreciable = Math.max(0, price - residual). Ajout d'une bannière d'avertissement visible dans l'onglet Amortissement lorsque cette condition est détectée.
C11 Contrat fantôme à la réouverture du wizard Restitution App.jsx
Problème
Après onDone(), restitutionContract n'était jamais remis à null. Réouvrir la restitution réaffichait le dernier contrat clôturé. Si restitutionContract était null, l'utilisateur était bloqué sans bouton retour.
Correction
onDone() appelle setRestitutionContract(null) avant navigation. Guard ajouté : si page === 'restitution' et contract est null, redirection automatique vers la liste des contrats.
C12 Nouvelle facture toujours créée lors d'une prolongation si dailyRate absent OtherPages.jsx
Problème
rateChanged = newRate !== Number(undefined) → NaN !== NaN → toujours true. Une nouvelle facture était systématiquement créée même sans aucun changement de tarif.
Correction
const originalRate = Number(contract.dailyRate) || 0 et guard originalRate > 0 dans la comparaison pour éviter la comparaison NaN et les factures fantômes.
C13 Classe CSS .badge-red manquante index.css
Problème
Plusieurs composants référençaient badge-red (contrats annulés, badges urgents) mais la classe n'existait pas dans le CSS. Ces éléments s'affichaient sans aucun style de coloration.
Correction
Ajout de .badge-red et .badge-purple dans index.css, cohérents visuellement avec les autres classes badge existantes dans le design system.
C14 Logique de calcul des frais dupliquée entre Step3 et Step4 (Restitution) Restitution.jsx
Problème
Le calcul des frais (km supplémentaires, carburant) était copié-collé identiquement dans Step3Damages et Step4Closure. Un changement de taux dans un endroit n'était pas répercuté dans l'autre.
Correction
Extraction dans une fonction pure partagée computeExtraFees({ vehicle, returnMileage, returnFuelLevel, contract, damageFee }). Les deux composants l'appellent désormais, garantissant la cohérence des calculs.

Avertissements Traités
Améliorations de robustesse, cohérence et maintenabilité
19 traités
Groupe A NewRental.jsx 3 avertissements traités
  • Draft localStorage : Photos (base64) exclues du draft pour éviter QuotaExceededError. Bloc try/catch ajouté dans saveDraft pour gérer les dépassements de quota gracieusement. Résolu
  • Validation dates : Message d'erreur inline si endDate < startDate. canContinue = false dans ce cas pour bloquer la progression du wizard. Résolu
  • Validation champs : Vérification renforcée dans RentalStep pour tous les champs obligatoires avant de permettre la validation de l'étape. Résolu
Groupe B storage.js 4 avertissements traités
  • Status facture par défaut : status: invoice.status || 'pending' ajouté dans saveInvoice pour garantir un statut initial cohérent à toutes les nouvelles factures. Résolu
  • Disponibilité véhicules : Filtre élargi — exclut les contrats non-closed et non-cancelled (pas seulement active) pour une gestion correcte des états intermédiaires. Résolu
  • Seed data enrichie : Contrat démo clôturé + contrat actif (v4) + facture payée ajoutés pour permettre des tests complets sur tous les états du cycle de vie d'un contrat. Résolu
  • Plaques format cohérent : 5 véhicules seed mis à jour au format pipe pour assurer la cohérence avec la fonction displayPlate. Résolu
Groupe C Fleet.jsx (pages/) 3 avertissements traités
  • configBanner nettoyé : configBanner remis à null dans openEdit pour éviter l'état fantôme qui affichait une bannière de configuration périmée lors de l'édition d'un autre véhicule. Résolu
  • Avertissement date d'achat estimée : Message visible si la date a été auto-remplie depuis vehicle.year plutôt que saisie manuellement, pour informer l'utilisateur du caractère estimatif. Résolu
  • RentalsTab réactif : useState(() => getContracts()) au lieu d'une lecture directe non-réactive, pour que les contrats affichés se mettent à jour sans rechargement de page. Résolu
Groupe D Restitution.jsx (pages/) 2 avertissements traités
  • Import getGeneralConfig supprimé : Import inutilisé retiré depuis la migration vers vehicle.maxKmEnabled, réduisant les dépendances et évitant la confusion lors des futures modifications. Résolu
  • Message d'erreur kilométrage : Affichage inline si returnMileage < startMileage, empêchant la saisie d'une valeur incohérente et guidant l'utilisateur vers la correction. Résolu
Groupe E OtherPages.jsx (pages/) 5 avertissements traités
  • Formatage montants : toLocaleString('fr-MA') appliqué à tous les montants MAD pour un affichage conforme aux conventions numériques marocaines. Résolu
  • Badge cancelled : Couleur corrigée de orange à rouge (badge-red) pour une distinction visuelle claire entre contrats annulés et autres statuts d'alerte. Résolu
  • Statut factures : Affichage 3 niveaux — Payée / En attente / fallback — pour couvrir tous les états possibles d'une facture et éviter l'affichage de valeurs brutes. Résolu
  • contracts réactif dans Clients : useState avec rechargement pour que la liste des contrats associés à un client se rafraîchisse correctement après modification. Résolu
  • invoices réactif dans Invoices : Même pattern useState appliqué à la liste des factures pour garantir la réactivité de l'interface après chaque opération CRUD. Résolu
Groupe F pdf.js 1 avertissement traité
  • displayPlate fallback : Retourne la plaque brute si aucun pipe n'est détecté, assurant la compatibilité avec les données existantes créées avant la migration au format pipe. Résolu
